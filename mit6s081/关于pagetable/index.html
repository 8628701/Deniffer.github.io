关于xv6的pagetable的一些想法

关于page table 的一些夹带私货

主要目的还是为了加深一下对page table的印象，从xv6来说的话，一个pte(page table entry)大小是64bit， 整个虚拟内存可以看成一个超大的数组，而我们可以通过数组下标来寻址(也就是我们说的虚拟地址，取地址的过程)，问题在于如何将虚拟地址和实实在在的物理地址进行映射呢？这是问题的关键，也是弄清楚pagetable 工作原理的关键。

首先关于物理地址(physical address)的前置知识

RISC-V的硬件决定了物理地址是一个56bits大小，因为只用了56根地址线。也就是说一个物理地址只需要56bits就可以表示，但是为了与虚拟地址统一，我们也用64bits来表示物理地址(在C里面为int64类型)，只是前面的64-56= 8bits 置为0.下面是一张很关键的图。阐述了虚拟内存以及物理内存的关系。 首先我们得先知道，物理地址是由硬件决定的。也就是说当xv6还没完全启动的时候，物理内存的一小部分就已经被写死了，比如io设备/网卡/等的物理设备所对应的地址就已经被写死了，但是我们可以通过 1. 直接访问物理地址(在最开始的时候是可以直接通过物理地址访问的，但是一旦mmu开始工作之后就必须使用虚拟地址的方法) 2. 虚拟地址->物理地址 的一一映射来访问或者说读写这些设备。

举一些例子来说明，比如当xv6的vm还没开始映射的时候,这时候只存在物理内存,假设你访问0x800000001,这其实应该是合法地址，因为这时候像异常/中断什么的都没有设置。但是你不知道0x80000001里面存的是什么，也许是00000000,也许是其他的东西，谁都不知道。但是你要明白这是合法的，但是这并没有什么用。 突出一个关键知识Virtual Memory 是Operating System 提供的一种mechanism，不是硬件提供的，但是硬件在里面也确实做了一点东西比如mmu

/*
 * create a direct-map page table for the kernel.
 */
void
kvminit()
{
  kernel_pagetable = (pagetable_t) kalloc();
  memset(kernel_pagetable, 0, PGSIZE);

  // uart registers
  kvmmap(UART0, UART0, PGSIZE, PTE_R | PTE_W);

  // virtio mmio disk interface
  kvmmap(VIRTIO0, VIRTIO0, PGSIZE, PTE_R | PTE_W);

  // CLINT
  kvmmap(CLINT, CLINT, 0x10000, PTE_R | PTE_W);

  // PLIC
  kvmmap(PLIC, PLIC, 0x400000, PTE_R | PTE_W);

  // map kernel text executable and read-only.
  kvmmap(KERNBASE, KERNBASE, (uint64)etext-KERNBASE, PTE_R | PTE_X);

  // map kernel data and the physical RAM we'll make use of.
  kvmmap((uint64)etext, (uint64)etext, PHYSTOP-(uint64)etext, PTE_R | PTE_W);

  // map the trampoline for trap entry/exit to
  // the highest virtual address in the kernel.
  kvmmap(TRAMPOLINE, (uint64)trampoline, PGSIZE, PTE_R | PTE_X);
}
  1. ‘freerange’函数将128MB挂到freelist，也就是说现在128MB内存基本都是空的(除了刚开始kernel.ld的部分被指定的那些)
  2. ‘kvminit’函数设置映射
  3. ‘kmalloc’函数从freelist中找到空闲的page，进行分配
  4. ‘kvmmap’ 函数开始映射，将pte 添加到page table里面
  5. ‘mappages’ 创建一个page所有的pte 从va开始->va+PAGESIZE(4096)
  6. 每一个pte都是32bits 占4Bytes

翻译小流程

当我们拿到一个进程的虚拟地址的时候，比如0xfffff，如何将它翻译成一个物理地址呢？

当你切换进程的时候，对应的satp寄存器也会随之指向该进程对应的页表，假设是和xv6一样的三级树状的页表结构，那么satp指向的就是L0的pagetable，这个pagetable 大小是4096Bytes，一个虚拟地址大小是8Bytes,也就是64bits，那么一个page table就可以存放512个虚拟地址，所以对应的虚拟地址L0part需要有9bits，来进行索引，通过这9bits，我们得到page directory的数组下标，通过索引pagetable[L0],获得了一个PPN(physical Page number),也就是L1所对应的page directory，通过sequence 9bits，我们又可以得到一个pagetable[L1]，得到一个PPN，这个PPN就是真正我们需要索引的L2的页表，随后我们就可以通过第三part的9bits，来找到pagetable[L2]（这里才是将address作为条目，之前都是将page作为条目），得到一个index，将index+offset，我们就可以translate into 对应的PA(physical address),这样做的坏处在于我们需要访存三次，才能获得一个真正的PA，所以需要page cache(TLB)来降低这种开销，值得一提的是MMU为我们做了这些事情.
